# Nymbus Supplemental Questions

## How have you used AI tools while building a project? What tools did you use? What did you use them for? What worked well?

I have used AI in two ways: as a builder of AI features and as a developer using AI to work faster. At SmartSolve, I use Codex and Claude Code to think through implementation plans, debug issues, review code, and iterate on features, but I still treat the final decision as my responsibility. One thing I cared about there was making AI-assisted work safer, so I helped create a sandboxed development setup for Claude Code around company source code instead of letting it run loosely in an inconsistent local environment. I also started thinking about AI work less as one chat window and more as an engineering workflow: running multiple agents in parallel in separate Docker containers, tmux sessions, or git worktrees so different approaches can be isolated, compared, and reviewed before anything is merged. On the product side, I have built with OpenAI API, Gemini API, Mistral AI, and LangChain through projects like RocketGrader, FalconGraph, and my Diff-Grounded PR Generation research. What has worked best for me is using AI to get unstuck, explore options, and speed up repetitive implementation work, while keeping the requirements, security, and code quality checks grounded in my own review.

## What signs would tell you that AI-generated code needs improvement, even if it seems correct at first glance?

This is honestly becoming a harder question to answer because AI-generated code can look very complete now, and sometimes reviewing it properly can take as much time as writing it yourself. That said, I usually start by looking for places where the code feels correct in isolation but disconnected from the rest of the system. For example, I watch for code that only handles the happy path, skips input validation, uses vague error handling, adds a dependency without a clear reason, or ignores the patterns already used in the project. I also look closely at security and data concerns, especially broad auth assumptions, unclear access control, silent failure modes, and AI outputs that are not grounded in the right source material. If I cannot explain why the code is correct, why it belongs in that part of the codebase, and how I would debug it later, I treat that as a sign it still needs work.

## What steps do you take to ensure your code is reliable and scalable?

I try to make reliability part of the way I build instead of something I check only at the end. I start by understanding the expected behavior, the data shape, the user flow, and the failure cases, then I build in smaller pieces that are easier to review and test. I validate inputs, handle errors directly, keep permissions scoped, and read through generated or unfamiliar code line by line before trusting it. At AAIS, this mindset showed up in production data work: profiling source tables, validating outputs, building repeatable Python/PySpark/AWS Glue workflows, partitioning S3 data movement, and using IAM controls around insurance data. For scalability, I avoid hard-coded assumptions, keep APIs and database interactions clear, separate concerns, and document workflows so the next engineer can understand the system without needing me in the room.

## Describe a time you improved something - a workflow, a project, or a tool - without being asked. Why did you do it? What was the impact?

At SmartSolve, I noticed that local development and AI-assisted coding could get messy quickly when different machines were using different setups across WSL2 and Colima. I also did not want Claude Code working directly in an uncontrolled environment around proprietary code, so I built a Docker-based devcontainer and sandboxed workflow for it. From there, I started treating AI-assisted development more like parallel engineering work: separate containers, tmux sessions, or git worktrees for different agents or approaches, then a deliberate review step before bringing anything back into the main codebase. I did it because small setup problems turn into real engineering friction: they slow down feature work, make bugs harder to reproduce, and make it less clear what the AI tool is allowed to touch. The impact was a cleaner and more repeatable development setup, less environment drift, and a safer foundation for building internal tools like the Next.js onboarding tracker.
